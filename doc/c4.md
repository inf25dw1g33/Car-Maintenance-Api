# Capítulo 4: Apresentação

## Demonstração

Durante a apresentação serão demonstrados os seguintes pontos:

### 1. Arquitetura Docker

- Execução do comando `docker-compose up` para iniciar os containers
- Verificação dos containers em execução com `docker ps`
- Demonstração da comunicação entre containers

### 2. Documentação Swagger

- Acesso à interface Swagger UI em `/docs`
- Navegação pelos endpoints disponíveis
- Execução de requests através da interface

### 3. Operações CRUD

Demonstração das operações em cada recurso:

- **Create**: Criação de novos registos
- **Read**: Listagem e consulta individual
- **Update**: Atualização de registos existentes
- **Delete**: Remoção de registos

### 4. Filtros HTTP

Exemplos de utilização dos filtros:

```bash
# Filtrar proprietários por nome
GET /api/owners?name=Maria

# Filtrar veículos por marca e ano
GET /api/vehicles?brand=BMW&year=2020

# Filtrar registos de serviço por data
GET /api/service-records?date_from=2024-01-01&date_to=2024-06-30
```

### 5. Relação M:N

Demonstração da relação muitos-para-muitos entre Service Records e Mechanics:

```bash
# Listar mecânicos de um serviço
GET /api/service-records/1/mechanics

# Atribuir mecânico a um serviço
POST /api/service-records/1/mechanics/5

# Remover mecânico de um serviço
DELETE /api/service-records/1/mechanics/5

# Listar serviços de um mecânico
GET /api/mechanics/1/service-records
```

### 6. Postman

- Importação da coleção disponibilizada
- Execução de requests de exemplo
- Demonstração dos diferentes endpoints

## Utilização de Inteligência Artificial

Neste trabalho foi utilizada a ferramenta **Claude** (Anthropic) como apoio ao desenvolvimento. De seguida documenta-se o processo de utilização.

### Contexto de Utilização

A IA foi utilizada como ferramenta de apoio em várias fases do desenvolvimento:

1. **Estruturação inicial do projeto**
   - Prompt: "Preciso de ajuda para estruturar uma API REST para gestão de manutenção de automóveis com Node.js e MySQL"
   - Contribuição: Sugestão da estrutura de pastas (controllers, services, api, utils) e das entidades do modelo de dados

2. **Especificação OpenAPI**
   - Prompt: "Ajuda-me a criar a especificação OpenAPI 3.0 para os endpoints da API com filtros"
   - Contribuição: Estruturação do ficheiro openapi.yaml com schemas, parâmetros de query e respostas

3. **Implementação dos filtros**
   - Prompt: "Como implementar filtros dinâmicos nas queries SQL de forma segura?"
   - Contribuição: Sugestão do padrão de construção de queries com parâmetros opcionais e prepared statements

4. **Relação M:N**
   - Prompt: "Qual a melhor forma de implementar a relação muitos-para-muitos entre service_records e mechanics?"
   - Contribuição: Criação da tabela associativa e endpoints para gestão da relação

5. **Configuração Docker**
   - Prompt: "Ajuda-me a configurar o Docker Compose para MySQL + Node.js com ambientes de dev e prod"
   - Contribuição: Estruturação dos ficheiros docker-compose com variáveis de ambiente e networking

6. **Documentação/Relatório**
   - Prompt: "Ajuda-me a fazer um relatório de acordo com o template disponibilizado pelo professor"
   - Contribuição: Criação da estrutura do relatório em Markdown seguindo o template

### Reflexão Crítica

A utilização da IA como ferramenta de apoio foi benéfica nos seguintes aspetos:

**Pontos positivos:**
- Acelerou a criação da estrutura base do projeto
- Ajudou a identificar boas práticas (ex: separação em camadas, prepared statements)
- Facilitou a criação de documentação consistente
- Sugeriu soluções para problemas específicos (filtros dinâmicos, relação M:N)

**Limitações identificadas:**
- Por vezes o código sugerido necessitou de ajustes para funcionar corretamente
- Foi necessário validar as sugestões e compreender o código antes de o utilizar
- Algumas configurações específicas do ambiente tiveram de ser corrigidas manualmente

**Aprendizagem:**
- A IA não substitui o conhecimento técnico, mas complementa-o
- É essencial compreender o código gerado e não apenas copiá-lo
- A validação e teste continuam a ser responsabilidade do programador
