# Capítulo 1: Apresentação do Projeto

## Tema

O tema escolhido para este trabalho prático foi o desenvolvimento de uma **API REST para Gestão de Manutenção de Automóveis**. Este sistema permite a gestão completa de uma oficina mecânica, incluindo o registo de proprietários, veículos, tipos de serviço, registos de manutenção e mecânicos.

## Descrição

A aplicação foi desenvolvida seguindo uma abordagem **Design-first**, onde a especificação OpenAPI foi criada antes da implementação do código. Isto permite uma documentação clara e consistente da API, facilitando o desenvolvimento e a integração com outros sistemas.

### Funcionalidades Principais

O sistema oferece as seguintes funcionalidades:

- **Gestão de Proprietários**: Registo e gestão de clientes da oficina, incluindo nome, email e telefone.
- **Gestão de Veículos**: Registo de veículos associados a proprietários, com informações como marca, modelo, matrícula e ano.
- **Tipos de Serviço**: Catálogo de serviços disponíveis na oficina (mudança de óleo, troca de pneus, revisões, etc.).
- **Registos de Manutenção**: Histórico completo de manutenções realizadas em cada veículo, incluindo data, quilometragem, custo e notas.
- **Gestão de Mecânicos**: Registo de mecânicos com as suas especializações.

### Filtros Implementados

A API disponibiliza filtros através de parâmetros HTTP para facilitar a pesquisa de recursos:

- **Proprietários**: Filtro por nome (pesquisa parcial)
- **Veículos**: Filtro por marca, ID do proprietário ou ano de fabrico
- **Tipos de Serviço**: Filtro por nome (pesquisa parcial)
- **Registos de Manutenção**: Filtro por ID do veículo, tipo de serviço ou intervalo de datas
- **Mecânicos**: Filtro por nome ou especialização

## Motivação

A escolha deste tema deve-se à sua relevância prática e ao facto de permitir demonstrar:

1. Uma estrutura de dados rica com múltiplas entidades relacionadas
2. Relações de cardinalidade 1:N (proprietário → veículos, veículo → registos)
3. Relação de cardinalidade M:N (registos de serviço ↔ mecânicos)
4. Operações CRUD completas em todos os recursos
5. Filtros úteis para pesquisa de dados

## Arquitetura

A aplicação segue uma arquitetura de três camadas:

```

│            Cliente (Swagger UI / Postman)           │
└─────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────┐
│              API REST (Node.js/Express)             │
│  ┌───────────┐  ┌───────────┐  ┌───────────┐        │
│  │  Routes   │→ │Controllers│→ │ Services  │        │
│  └───────────┘  └───────────┘  └───────────┘        │
└─────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────┐
│              Base de Dados (MySQL)                  │
└─────────────────────────────────────────────────────┘
```
